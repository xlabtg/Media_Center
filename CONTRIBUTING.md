# Руководство для участников НМЦ

Спасибо за вклад в платформу «Народный Медиацентр» (НМЦ). Документ описывает,
как устроена работа над проектом: планирование, ветки, коммиты, стандарты кода,
тестирование и процесс ревью.

> Рабочий язык проекта — **русский**. Идентификаторы, код и команды остаются
> в оригинальном виде. Документация, issue, описания PR и коммиты — на русском.

---

## 1. Как устроен план

Разработка декомпозирована на **этапы → эпики → задачи**:

- **Этапы** — это [milestones](https://github.com/xlabtg/Media_Center/milestones) (0–8) и метки `stage:*`.
- **Эпики и задачи** — issue с метками `type:*`, `priority:*`, `area:*`, `component:*`.
- Точка входа — 🗺 [Мастер-план #104](https://github.com/xlabtg/Media_Center/issues/104).
- Детальная декомпозиция — [docs/DEVELOPMENT_PLAN.md](docs/DEVELOPMENT_PLAN.md).
- Сроки и цели этапов — [docs/ROADMAP.md](docs/ROADMAP.md).

Перед началом работы найдите соответствующий issue (или создайте его по шаблону)
и привяжите к нему ветку и PR.

### Таксономия меток

| Группа | Значения |
|--------|----------|
| `type:` | `epic`, `feature`, `task`, `research`, `docs`, `chore`, `test`, `bug` |
| `priority:` | `critical`, `high`, `medium`, `low` |
| `stage:` | `0-discovery` … `8-scale-ops` |
| `area:` | `backend`, `frontend`, `devops`, `ai-ml`, `data`, `security`, `compliance`, `design`, `qa`, `product` |
| `component:` | модуль системы (`contribution-ledger`, `cglr`, `hitl-payout`, …) |

## 2. Ветки и коммиты

- Основная ветка — `main`. **Прямые пуши в `main` запрещены.**
- Работайте в ветках вида `issue-<номер>-<краткое-описание>` или
  `feature/<краткое-описание>`.
- Для исправлений и инфраструктурных задач допустимы ветки
  `fix/<краткое-описание>` и `chore/<краткое-описание>`, если они привязаны к
  issue в PR.
- Изменения вносятся через **Pull Request** в `main`.
- Перед открытием PR и перед финальным ревью влейте свежий `main` в свою ветку.
- PR должен закрывать или ссылаться на issue (`Closes #<номер>`,
  `Fixes #<номер>` или `Refs #<номер>`).

### Conventional Commits

Коммиты пишутся в формате **Conventional Commits** с русским описанием:

```text
<тип>(<scope>): <краткое описание>
```

`scope` опционален и обычно совпадает с сервисом, библиотекой или областью
изменения: `api-gateway`, `contribution-ledger`, `shared`, `infra`, `docs`.
Допустимые типы: `feat|fix|docs|test|refactor|chore|ci|perf`.

Краткое описание пишется в повелительном наклонении и без точки в конце:

```text
chore(infra): добавь базовую структуру монорепозитория
docs(contribution-ledger): уточни границы сервиса учёта вклада
test(shared): добавь проверку tenant-aware ошибок
```

Если изменение ломает совместимость будущего API или контракта, укажите это в
теле коммита через `BREAKING CHANGE:` и опишите миграционный путь. Связывайте
коммиты и PR с issue (`#<номер>`).

## 3. Стандарты кода

Технологический стек baseline: **Python 3.13.x**, FastAPI 0.137.2,
Pydantic 2.13.4, SQLAlchemy 2.0.51 (async, asyncpg 0.31.0), Alembic 1.18.4,
pytest 9.1.0. Полная матрица версий зафиксирована в
[ADR-0006](docs/adr/0006-technology-stack-and-versions.md).

- **Форматирование и линт:** `ruff` (lint + format).
- **Типизация:** `mypy` (строгий режим для новых модулей).
- **Архитектура:** микросервисы используют общую библиотеку `shared`
  (модели, `audit_logger`, утилиты тенанта, ошибки).
- **Мультитенантность:** каждый запрос и запись содержат `tenant_id`;
  межтенантный доступ обязан возвращать `403 tenant_isolation_violation`.

### Перед коммитом (локальный CI)

```bash
ruff check .
ruff format --check .
mypy .
pytest
```

Рекомендуется установить `pre-commit`, чтобы проверки запускались автоматически.

## 4. Тестирование

- Покрывайте бизнес-логику unit-тестами (с моками внешних систем).
- Интеграционные и e2e-тесты — через общий фреймворк проекта.
- **Обязательны тесты мультитенантной изоляции** (cross-tenant → `403`).
- Исправление бага сопровождается тестом, воспроизводящим проблему.
- Задавайте разумные таймауты на тестах, чтобы сбои выявлялись быстро.

---

## 5. Безопасность и комплаенс (обязательно)

- **Никогда** не коммитьте секреты — только [`.env.example`](.env.example).
- В блокчейн-аудит пишутся **только SHA256-хэши и метаданные** — без денежных
  сумм и персональных данных (ПДн).
- **HITL:** AI исполняет, но не принимает финальных решений — выплаты и
  критичные действия контролирует Совет (вето + 2FA).
- Соблюдайте требования по ПДн (ФЗ-152) и ToS площадок (см.
  [docs/COMPLIANCE.md](docs/COMPLIANCE.md), [docs/SECURITY.md](docs/SECURITY.md)).

---

## 6. Процесс Pull Request

1. Создайте ветку от свежего `main`.
2. Внесите изменения, добавьте тесты и обновите документацию.
3. Прогоните локальный CI (раздел 3).
4. Откройте PR по [шаблону](.github/pull_request_template.md), укажите
   связанный issue (`Closes #<номер>`).
5. Дождитесь зелёного CI и пройдите ревью.
6. После одобрения PR вливается в `main` (без прямых merge в обход PR).

Спасибо за вклад! 🎯
