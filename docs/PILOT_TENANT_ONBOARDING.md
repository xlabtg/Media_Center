# Пилотный tenant и онбординг

Дата фиксации: 2026-06-20.

Статус: launch packet для issue #91.

Документ описывает подготовку pilot tenant `nmc-pilot`, регистрацию и
онбординг первой группы. Источник истины для локальной проверки -
`infra/local/fixtures/pilot-tenant.json`; контракт проверяется тестом
`tests/test_pilot_tenant_issue91_acceptance_contract.py`.

## 1. Состав tenant

Пилот настроен как один tenant без реальных ПДн и внешних секретов:

| Параметр | Значение |
|----------|----------|
| Tenant slug | `nmc-pilot` |
| Статус | `pilot_ready` |
| Участники | 20 синтетических участников |
| Приемочный диапазон | 15-25 участников |
| Политика данных | без ПДн, только synthetic handles и hash refs |
| Окно запуска | 2026-06-22 - 2026-06-26 |

Распределение ролей фиксирует полный RBAC-контур пилота:

| Роль | Количество | Назначение |
|------|------------|------------|
| `council` | 6 | Вето, стратегические решения, кураторство |
| `presidium` | 2 | Надзор, аудит, разбор спорных случаев |
| `board` | 3 | Операционное управление и поддержка онбординга |
| `member_full` | 5 | Действительные пайщики, вклад и голос |
| `member_assoc` | 4 | Ассоциативные пайщики, входящий поток участников |

## 2. Онбординг 12-36 часов

Целевое окно онбординга - 24 часа, допустимый диапазон - 12-36 часов. Участник
считается зарегистрированным, когда создан tenant-scoped профиль, назначена
роль и записан `registered` status. Участник считается находящимся в онбординге,
когда назначен куратор и присутствуют все обязательные шаги:

| Шаг | Владелец | SLA |
|-----|----------|-----|
| `profile` | Правление | 4 часа |
| `consents` | Правление | 4 часа |
| `channels` | Notification Gateway | 8 часов |
| `mentor_intro` | Совет | 12 часов |
| `council_review` | Совет | 24 часа |

Необязательный шаг `first_task` открывается после ручной проверки Совета. В
фикстуре участники распределены по статусам `scheduled`, `in_progress` и
`ready_for_review`, чтобы pilot dashboard видел и очередь, и готовые анкеты.

## 3. Роли и пороги Совета

Порог стратегических решений Совета - 2/3. HITL-окно вето для пилота задано как
8 часов при допустимых границах 4-12 часов. Выплаты и изменение порогов требуют
2FA и ручного подтверждения:

| Операция | Порог |
|----------|-------|
| `payout.execute` | 2 подтверждения и 2FA |
| `policy.threshold.update` | 2 подтверждения и 2FA |
| Изменение политик | 3 подтверждения, XAI-обоснование и audit hash |
| Выпуск контента | обязательный draft review, без автопубликации |

Все изменения порогов пишутся в `blockchain-auditor` как hash-only metadata.

## 4. Runbook запуска

1. Применить локальные миграции и сиды: `make migrate`.
2. Сверить `nmc-pilot` с `infra/local/fixtures/pilot-tenant.json`.
3. Проверить, что все 20 synthetic handles имеют tenant id
   `00000000-0000-4000-8000-000000000001`.
4. Назначить куратора из `council-01` - `council-06`.
5. Открыть онбординг в Web Cabinet и проверить шаги `profile`, `consents`,
   `channels`, `mentor_intro`, `council_review`.
6. В панели Совета проверить порог 2/3, окно вето 8 часов и 2FA для
   чувствительных операций.
7. Включить KPI monitoring tenant dashboard только после ручного go/no-go.

## 5. KPI и rollback

Пилот стартует с целями из `docs/ROADMAP.md`: 20-30 материалов в неделю,
10 000+ просмотров в неделю, среднее чтение больше 3 минут, 50+ комментариев в
неделю, 10+ задач в неделю и 1-2 инициативы в месяц.

Rollback выполняется как отключение tenant features без удаления audit history:

- поставить tenant status в `pilot_paused`;
- остановить внешние публикации и выплаты;
- сохранить hash-only audit trail;
- выгрузить обезличенную ретроспективу для Совета;
- не переносить synthetic handles в production без отдельной проверки ПДн.
