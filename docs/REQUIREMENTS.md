# Требования и границы MVP НМЦ

Документ фиксирует базовую версию функциональных требований (FR),
нефункциональных требований (NFR), границ MVP и KPI пилота для issue
[#3](https://github.com/xlabtg/Media_Center/issues/3).

Статус: baseline для согласования с заказчиком. После утверждения изменения в
границах MVP фиксируются отдельными issue, ADR или обновлением этого документа.

## 1. Источники и область применения

### 1.1. Источники

- Исходный регламент: [Народный Медиацентр Регламент.mhtml](../Народный%20Медиацентр%20Регламент.mhtml).
- Продуктовое видение: [VISION.md](VISION.md).
- Дорожная карта: [ROADMAP.md](ROADMAP.md).
- Архитектура: [ARCHITECTURE.md](ARCHITECTURE.md).
- Экономика: [ECONOMICS.md](ECONOMICS.md).
- Управление: [GOVERNANCE.md](GOVERNANCE.md).
- Безопасность и право: [SECURITY.md](SECURITY.md), [COMPLIANCE.md](COMPLIANCE.md).

### 1.2. Цель MVP

Проверить на одном пилотном тенанте, что НМЦ может поддерживать полный
контур: участники создают и распространяют контент, система учитывает вклад,
Совет контролирует чувствительные действия, а KPI пилота собираются достаточно
прозрачно для решения о масштабировании.

### 1.3. Пилотный контур

| Параметр | Значение для MVP |
|----------|------------------|
| Тенанты | 1 пилотный tenant |
| Размер ядра/Совета | 15-25 активных участников |
| Срок ориентира | 6-8 недель разработки до пилота |
| География и право | РФ, с учетом ФЗ-152, ФЗ-3085-1, ФЗ-149/436 |
| Приоритет площадок | Telegram, VK, Dzen, OK; расширение через реестр платформ |
| Принцип управления | Human-in-the-Loop: ИИ исполняет, человек задает правила, утверждает и накладывает вето |

## 2. Пользователи и роли

| Роль | Назначение в MVP |
|------|------------------|
| Аудитория | Читает публичный контент, задает вопросы, может перейти в онбординг. |
| Ассоциативный пайщик | Выполняет задачи, получает совещательный голос и видит свой вклад. |
| Действительный пайщик | Имеет решающий голос, доступ к закрытым ресурсам и участие в распределении благ. |
| Совет | Утверждает стратегические правила, пороги, вето, доступ к аудиту и спорные решения. |
| Президиум Совета | Выполняет надзор, аудит и разрешение споров. |
| Правление/администратор | Управляет операционными настройками, задачами, публикациями и выплатными очередями. |
| ИИ-агенты | Готовят черновики, анализируют, маршрутизируют, но не принимают финальные чувствительные решения. |

## 3. Функциональные требования

### FR-01. Тенант и идентификация

Система обязана поддерживать один пилотный tenant с обязательным `tenant_id` в
JWT, API-запросах, записях БД, событиях, логах, метриках, payout-экспортах и
аудит-хэшах.

Критерий приемки: любой запрос к ресурсу другого tenant возвращает `403
tenant_isolation_violation` и записывается в аудит.

### FR-02. Регистрация, онбординг и согласия

Система обязана поддерживать регистрацию участника с базовыми каналами связи
(телефон, Telegram, email) и явными согласиями на обработку данных. Подключение
социальных профилей выполняется только после подтверждения участником.

Для MVP допускается полуавтоматический онбординг: ИИ формирует черновик анкеты
и рекомендацию статуса, а Правление или Совет подтверждает решение.

Критерий приемки: у каждого участника есть статус, согласия, tenant-привязка и
история изменения статуса.

### FR-03. Роли, RBAC и статусы пайщиков

Система обязана различать роли `council`, `presidium`, `board`,
`member_full`, `member_assoc`, `audience`. Переход из ассоциативного в
действительный статус не должен происходить полностью автоматически: требуется
решение органа управления по правилам регламента.

Критерий приемки: операции Совета, выплат, аудита и настройки политик доступны
только ролям с соответствующими правами.

### FR-04. Учет вклада

Система обязана фиксировать события вклада: идея, создание контента,
публикация, усиление, вовлечение, администрирование, координация, экспертная
консультация, реферальная конверсия и недельная активность.

Базовая формула:

```text
final_points = round(base * platform_mult * reach_mult * amp_mult, 2)
```

Критерий приемки: каждое событие имеет `tenant_id`, тип, автора, начисленные
баллы, метаданные, timestamp и `audit_hash`.

### FR-05. Вес вклада и доля выплаты

Система обязана рассчитывать вес вклада (Кв) и долю выплаты:

```text
kv = min(points_member / avg_points_council, 0.10)
payout_share = kv_capped / sum(kv_capped)
```

Критерий приемки: ни один участник не может получить Кв выше 0.10, а экспорт
долей для выплат содержит только данные своего tenant.

### FR-06. Экономические правила и фонды

MVP обязан поддержать конфигурируемые правила распределения 35/17/48 и фонды
участников, описанные в [ECONOMICS.md](ECONOMICS.md). Значения коэффициентов,
фондов и мультипликаторов должны задаваться политиками, а не быть зашитыми в
код без возможности изменения Советом.

Критерий приемки: Совет может изменить параметры через утвержденную политику,
а изменение логируется и попадает в аудит.

### FR-07. Генерация и маршрутизация контента

Система обязана поддерживать генерацию/подготовку контента по утвержденным
темам, шаблонам и тематическим дням, а также маршрутизацию ссылок L1/L2/L3:

- L1: главный CTA, заданный администратором;
- L2: ссылка автора или профиля tenant;
- L3: ротационный партнер по весу вклада.

Критерий приемки: готовый материал валидируется под выбранную площадку и
передает событие вклада в Contribution Ledger.

### FR-08. Публикация на площадках

MVP обязан поддержать публикацию или подготовку публикации минимум на трех
приоритетных площадках из набора Telegram, VK, Dzen, OK. Архитектура адаптера
должна позволять расширение до top-10 РФ и далее до полного реестра платформ.

Критерий приемки: публикация возвращает статус, platform id, ссылку/идентификатор
поста, ошибку при сбое и событие для учета вклада.

### FR-09. HITL-выплаты и право вето

Все выплаты и чувствительные операции обязаны проходить через HITL-контур:
очередь, уведомление Совета/Правления, окно вето 4-12 часов (по умолчанию 8),
2FA для подтверждения и обязательное обоснование при вето.

Критерий приемки: операция не исполняется раньше окончания правила HITL, если
для нее требуется подтверждение или открыто окно вето.

### FR-10. Блокчейн-аудит

Система обязана писать в приватный блокчейн только SHA256-хэши и технические
метаданные: `tenant_id`, `event_id`, тип события, статус, timestamp. Суммы,
ФИО, контакты, тексты персональных данных и платежные реквизиты в блокчейн не
попадают.

Критерий приемки: любую аудит-запись можно проверить по исходному payload без
раскрытия ПДн и денежных сумм.

### FR-11. Activity Command Center

Для Совета/администратора в MVP нужен минимальный командный центр активности:
список участников, статус недельной активности, задачи, просрочки, кампании,
пороговые настройки и журнал AI-действий.

Критерий приемки: Совет видит, кто выполнил минимум активности, кто требует
уведомления, какие задачи назначены и какие AI-действия ожидают подтверждения.

### FR-12. AI-автоматизация под контролем политик

ИИ-агенты могут выполнять подготовку контента, аналитику, рекомендации,
черновики ответов и ограниченные действия с площадками только при opt-in
пользователя и в пределах политик Совета: лимиты частоты, белые/черные списки,
режим предпросмотра, ручное подтверждение спорных действий.

Критерий приемки: массовые рассылки, автоответы и действия от имени участника
имеют явное согласие, лимит и аудит.

### FR-13. Голосовые данные и Voice-to-Chain

Если в MVP включается голосовой контур, исходное аудио хранится на стороне
клиента или в защищенном временном хранилище, транскрибируется локально или в
защищенном режиме, подтверждается пользователем и удаляется в течение 24 часов
после успешной обработки.

Критерий приемки: в блокчейн попадает только хэш транскрипта, хэш согласия и
метаданные события.

### FR-14. KPI и аналитика

Система обязана собирать KPI участия, контента, вовлеченности, действий,
HITL-операций, публикаций и технической нагрузки. В MVP достаточно агрегатов
для Совета и экспорта для ретроспективы пилота.

Критерий приемки: KPI доступны за период, фильтруются по tenant и не содержат
ПДн сверх необходимого минимума.

### FR-15. Уведомления

Система обязана отправлять уведомления о задачах, просрочках активности,
HITL-окнах, вето, выплатных очередях, публикационных ошибках и важных
изменениях политик.

Критерий приемки: уведомление содержит ссылку на действие, tenant-контекст и
не раскрывает данные другого участника без права доступа.

## 4. Нефункциональные требования

| ID | Требование | Приемка для MVP |
|----|------------|-----------------|
| NFR-01 | Мультитенантность по умолчанию | `tenant_id` обязателен во всех слоях; cross-tenant доступ возвращает `403 tenant_isolation_violation`. |
| NFR-02 | Безопасность | JWT HS256, RBAC, 2FA для чувствительных операций, TLS 1.3+, AES-256 для данных в покое. |
| NFR-03 | Приватность и ФЗ-152 | Явные согласия, минимизация ПДн, API удаления данных, авто-удаление сырого аудио за 24 ч. |
| NFR-04 | Комплаенс контента | Проверки и маркировка под ФЗ-149/436; отдельная юрпроверка до пилота. |
| NFR-05 | Аудитируемость | Все чувствительные операции имеют структурный лог и SHA256-аудит-хэш. |
| NFR-06 | HITL и объяснимость | ИИ-решения, влияющие на деньги, статусы, массовые действия и политики, объяснимы и подтверждаются человеком. |
| NFR-07 | Производительность | CGLR: 100 req/s при < 200 мс; Ledger: 50 событий/с; HITL: 10 очередей/ч, вето < 5 с; Messenger: 200 публикаций/мин при > 99% успеха. |
| NFR-08 | Надежность | Идемпотентные операции, ретраи публикаций, очереди для длительных действий, журнал ошибок. |
| NFR-09 | Наблюдаемость | Метрики, структурные логи и трассировка с `tenant_id`, без ПДн в логах. |
| NFR-10 | Расширяемость | Новые площадки, политики, коэффициенты и сервисы добавляются без переписывания ядра. |
| NFR-11 | Локальная разработка | Репозиторий должен поддерживать локальный запуск и проверки через docker-compose/CI по мере появления кода. |
| NFR-12 | Рабочий язык | Документация, issue, PR и пользовательские тексты ведутся на русском; код и идентификаторы остаются на английском. |

## 5. Границы MVP

### 5.1. Входит в MVP

| Область | Минимальный результат |
|---------|-----------------------|
| Tenant foundation | Один tenant, JWT с `tenant_id`, RBAC, аудит cross-tenant отказов. |
| Участники | Регистрация, статусы, согласия, базовый профиль, история статуса. |
| Contribution Ledger | Матрица баллов, мультипликаторы, Кв с потолком 0.10, экспорт payout shares. |
| CGLR | Шаблоны, тематические дни, L1/L2/L3 ссылки, валидация под площадку. |
| Messenger Adapter | Рабочий контур минимум для трех приоритетных площадок; расширяемый реестр. |
| HITL Payout Gateway | Очередь выплат, окно вето 4-12 ч, 2FA, лог причин вето. |
| Blockchain Auditor | SHA256-хэши и метаданные в приватном audit-контуре, без ПДн и сумм. |
| Activity Command Center | Дашборд активности, задачи, просрочки, политики AI-действий, журнал. |
| Analytics | KPI пилота по участию, контенту, вовлеченности, действиям и технической нагрузке. |
| Notifications | Уведомления по задачам, HITL, ошибкам публикаций и изменениям политик. |
| Compliance baseline | Согласия, минимизация ПДн, API удаления, контентные ограничения, юрчек-лист перед пилотом. |

### 5.2. Не входит в MVP и переносится в backlog

| Область | Причина переноса |
|---------|------------------|
| Многотенантная эксплуатация N tenant'ов | Требует SRE, SLA, backup/DR и отдельного этапа масштабирования. |
| Полная поддержка 102 площадок | Для пилота достаточно приоритетных РФ-площадок и расширяемого реестра. |
| Полностью автономные массовые рассылки и инвайтинг | Высокие ToS, ПДн и репутационные риски; нужны лимиты, юрпроверка и opt-in. |
| RL-KPI loop в проде | Допускается сбор данных; автоматическое обучение и изменение политик переносится на этап эксплуатации. |
| Полноценные мобильные приложения | MVP может использовать веб-кабинет, Telegram-клиент или адаптивный интерфейс. |
| Публичный блокчейн или криптовалюта | МСЦ остается внутренней метрикой; блокчейн только приватный audit-контур. |
| Автоматическое принятие в действительные пайщики | По регламенту требуется решение органа управления. |
| Полная бухгалтерия и налоговые отчеты | Нужна отдельная юридическая и бухгалтерская постановка задачи. |
| Продвинутый BI и предиктивная аналитика | На пилоте нужны измеримые KPI и экспорт, не полноценная BI-платформа. |
| Полное покрытие anti-blocking | Базовые прокси/ретраи допустимы, но агрессивный обход ограничений требует риск-анализа. |

## 6. KPI пилота

### 6.1. Продуктовые KPI

| Категория | Метрика | Цель пилота | Источник данных |
|-----------|---------|-------------|-----------------|
| Участие | Активные члены Совета/ядра | 15-25 | Activity Command Center, RBAC |
| Участие | Новые участники | +3-5 в месяц | Онбординг, реестр участников |
| Участие | Недельная активность ассоциативных пайщиков | >= 1 действие/неделю на активного участника | Contribution Ledger |
| Контент | Материалы | 20-30 в неделю | CGLR, Messenger Adapter |
| Контент | Публикация на площадках | >= 3 площадки для ключевых кампаний | Messenger Adapter |
| Контент | Просмотры | 10 000+ в неделю | Analytics Engine, площадки |
| Контент | Среднее время чтения | > 3 минут | Analytics Engine |
| Вовлеченность | Комментарии | 50+ в неделю | Analytics Engine, площадки |
| Действия | Задачи | 10+ в неделю | Activity Command Center |
| Действия | Инициативы | 1-2 в месяц | Совет/Правление, трекер задач |

### 6.2. Операционные и технические KPI

| Категория | Метрика | Цель пилота | Источник данных |
|-----------|---------|-------------|-----------------|
| Изоляция | Межтенантные утечки | 0 | Тесты, аудит, security checks |
| Аудит | Чувствительные операции с audit hash | 100% | Blockchain Auditor, audit log |
| Приватность | ПДн или суммы в блокчейне | 0 случаев | Проверка payload и chain records |
| Голос | Удаление сырого аудио | <= 24 ч после обработки | Voice-to-Chain logs |
| HITL | Вето/подтверждение доступно Совету | < 5 с на действие интерфейса/API | HITL Gateway metrics |
| HITL | Очереди выплат | >= 10 очередей/ч | HITL Gateway load test |
| Ledger | Обработка событий вклада | >= 50 событий/с | Load test |
| CGLR | Генерация/валидация | >= 100 req/s при p95 < 200 мс | Load test |
| Messenger | Публикации | >= 200 публикаций/мин, > 99% успеха | Adapter metrics |
| Уведомления | Доставка критичных HITL-уведомлений | >= 99% без потери событий | Notification Gateway |

## 7. Основные пользовательские сценарии MVP

1. Участник регистрируется, дает согласия, подключает базовые каналы связи,
   получает ассоциативный статус и выполняет первую задачу.
2. Совет утверждает тему и шаблон, CGLR готовит материал, адаптер публикует
   его на нескольких площадках, а Ledger начисляет вклад.
3. Участник накапливает баллы, система считает Кв с потолком 0.10 и формирует
   экспорт долей для выплат.
4. HITL Gateway открывает окно вето, Совет подтверждает или отклоняет выплату
   с 2FA, а audit-контур фиксирует хэш операции.
5. Совет видит KPI пилота, активность участников, просрочки и публикационные
   ошибки, после чего корректирует политики или задачи.
