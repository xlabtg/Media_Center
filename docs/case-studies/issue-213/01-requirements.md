## B. Функциональные требования контейнеризации (из чата Qwen)

Источник — дословные требования пользователя (см. `sources/qwen-chat-transcript.md`, Часть 1).

### REQ-1. Multi-stage build + минимальный runtime
- **REQ-1.1** Код собирается в builder-стадии; в финальный образ переносится только готовая программа.
- **REQ-1.2** В финальном образе нет компиляторов, отладчиков, build-инструментов («без дебагиров, компиляторов и прочего»).
- **REQ-1.3** Финальный образ — минимальный runtime.

### REQ-2. Готовый entrypoint
- **REQ-2.1** У контейнера есть готовый entrypoint, который при старте сам запускает приложение без ручных шагов.

### REQ-3. Единый порт 7700
- **REQ-3.1** Все сервисы слушают один и тот же порт — **7700**.

### REQ-4. Единый `/health`
- **REQ-4.1** Все сервисы отвечают на одинаковый endpoint healthcheck.

### REQ-5. Информационный `/info`
- **REQ-5.1** Endpoint содержит дату и время сборки.
- **REQ-5.2** Содержит версию компилятора/рантайма (для Python — версию интерпретатора).
- **REQ-5.3** Содержит commit hash.
- **REQ-5.4** Содержит название сервиса.
- **REQ-5.5** Содержит «реальную версию» из тегов репозитория (пример формата `01.04.15`); версии задаются git-тегами и используются для версионирования.

### REQ-6. Endpoint метрик
- **REQ-6.1** Все сервисы предоставляют endpoint метрик (Prometheus). Частично уже есть — нужно унифицировать.

### REQ-7. Логи в консоль
- **REQ-7.1** Все сервисы выводят логи в консоль (stdout).

### REQ-8. Управление уровнем логирования
- **REQ-8.1** Есть endpoint смены log level в runtime.
- **REQ-8.2** Уровень по умолчанию — INFO.
- **REQ-8.3** По умолчанию без request-лога (access-логи выключены).
- **REQ-8.4** Уровень логирования также задаётся при старте (через конфиг/переменную окружения).

### REQ-9. Публикация в GHCR
- **REQ-9.1** Все готовые контейнеры публикуются в GitHub Container Registry.

### REQ-10. Service-to-service авторизация (fallback chain)
- **REQ-10.1** Авторизация между сервисами через Kubernetes Service Account.
- **REQ-10.2** Если Kubernetes API недоступен — fallback на RSA-ключ (путь к точке монтирования передаётся через переменную окружения или файл конфигурации).
- **REQ-10.3** Последний fallback — общий секрет (shared secret).

### REQ-11. Единая структура артефакта
- **REQ-11.1** Артефакт сервиса лежит во всех контейнерах по одному и тому же пути.
- **REQ-11.2** Все необходимые файлы — в папке конфигурации рядом с артефактом.

### REQ-12. Non-root и безопасность
- **REQ-12.1** Контейнер и сервис запускаются от непривилегированного пользователя.
- **REQ-12.2** (Развитие из ответа источника) Hardening: read-only rootfs, `no-new-privileges`, drop capabilities, корректная init-система (tini) для обработки сигналов.
