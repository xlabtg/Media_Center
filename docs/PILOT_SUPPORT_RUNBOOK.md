# Runbook поддержки пилота и баг-фикс

Дата фиксации: 2026-06-20.

Статус: pilot-ready для issue #94.

Документ фиксирует операционный контур поддержки пилота `nmc-pilot`: прием
обращений, триаж, приоритизацию дефектов, выпуск исправлений и контроль через
CI. Источник истины для локальной проверки -
`infra/local/fixtures/pilot-support-queue.json`; контракт проверяется тестом
`tests/test_pilot_support_issue94_acceptance_contract.py`.

Поддержка пилота не принимает реальные ПДн, токены, коды 2FA, платежные
реквизиты, суммы выплат, закрытые материалы и сырой голос в issue, PR,
fixtures, логах и скриншотах. Для обращения достаточно tenant `nmc-pilot`, роли,
сценария, шага, времени, ожидаемого результата, фактического результата и
обезличенного описания влияния.

## 1. Критерии приемки #94

| Критерий | Как выполняется | Проверка |
|----------|-----------------|----------|
| Обращения обрабатываются в срок | Для каналов `support-intake`, `security-privacy` и `council-escalation` заданы SLA ответа; fixture фиксирует фактическое время первой реакции по каждому обращению. | `tests/test_pilot_support_issue94_acceptance_contract.py` |
| Критические дефекты устраняются приоритетно | Severity matrix выделяет P0/P1, требует эскалации, CI и короткий fix SLA; очередь содержит P0 `tenant_isolation` и P1 кейсы. | `infra/local/fixtures/pilot-support-queue.json` |
| Изменения проходят через CI | Release gate требует локальные проверки, workflow `CI`, security scan и сборку сервисных образов перед выпуском исправления. | `.github/workflows/ci.yml`, PR #194 |

## 2. Каналы поддержки

| Канал | Владелец | SLA ответа | Назначение |
|-------|----------|------------|------------|
| `support-intake` | `board` | 4 часа | Первый контакт для онбординга, задач, баллов, уведомлений, ошибок UI и FAQ. |
| `security-privacy` | `security` | 1 час | Tenant isolation, ПДн, секреты, token leak, 2FA, признаки утечки или обхода HITL. |
| `council-escalation` | `council` | 4 часа | Вето, go/no-go, спорные роли, политики, массовые действия и юридический риск. |
| `ci-release-gate` | `qa` | 4 часа | Проверка исправлений, локальных команд, CI и rollback перед выпуском. |

Первичный intake обязан присвоить обращению `case_id`, `tenant_slug`, категорию,
приоритет, владельца, `opened_at`, `first_response_at`, канал эскалации,
`evidence_policy` и признак `ci_required`. Если обращение содержит ПДн или
секреты, поддержка переносит только обезличенное описание и удаляет опасное
вложение по security/privacy процедуре.

## 3. SLA и приоритеты

| Приоритет | Примеры | SLA ответа | SLA исправления | Обязательная эскалация |
|-----------|---------|------------|-----------------|------------------------|
| P0 | tenant leak, token leak, ПДн-инцидент, обход HITL/2FA, массовый сбой доступа | 1 час | 24 часа | Security + Совет |
| P1 | несколько участников заблокированы, критичное уведомление не доставлено, срочное veto | 4 часа | 48 часов | Правление + Совет |
| P2 | один участник заблокирован, есть workaround, KPI или FAQ требуют уточнения | 8 часов | 5 дней | По решению владельца |
| P3 | документация, косметика, вопрос без влияния на запуск | 24 часа | 10 дней | Не требуется |

P0 и P1 всегда получают `ci_required=true`, связанный bugfix record и выпуск
через Pull Request. P0 разбирается первым даже при наличии более старых P2/P3
обращений. Для tenant isolation рабочий порядок один: остановить сценарий,
зафиксировать redacted evidence, открыть `security-privacy`, уведомить Совет и
не продолжать пилотный запуск до triage-решения.

## 4. Триаж дефектов

Триаж выполняется в четыре шага:

1. Классифицировать категорию: `tenant_isolation`, `onboarding`,
   `notification`, `hitl_payout`, `publication_review`, `faq` или новая
   категория с владельцем.
2. Выбрать P0-P3 по severity matrix и проверить, нужен ли Совет.
3. Назначить владельца: `security`, `qa`, `board`, `support` или `council`.
4. Зафиксировать workaround, тест воспроизведения, release gate и rollback.

Критерий готовности к исправлению: есть воспроизводимый тест, известен tenant
impact, нет ПДн/секретов в evidence, указан rollback и понятен набор локальных
проверок. Для UI-дефекта дополнительно прикладывается обезличенный скриншот
до/после, если это не раскрывает закрытые материалы.

## 5. Выпуск исправлений

Для каждого bugfix из P0/P1:

1. Добавить или обновить тест, который воспроизводит дефект.
2. Исправить минимальный контур без удаления существующих возможностей пилота.
3. Запустить локально:

```bash
ruff check .
ruff format --check .
black --check .
mypy .
pytest
bash experiments/validate_issue9_ci.sh
```

4. Обновить PR #194 с описанием: как воспроизвести, какой тест защищает
   исправление, какой rollback предусмотрен.
5. Дождаться workflow `CI`: `Lint, types, tests`, `Security scan` и `Build
   service image`.
6. После выпуска держать monitoring window 24-48 часов, а для P0 сохранять
   отдельную запись в ретроспективе Совета.

Если CI падает, поддержка скачивает свежие логи в `ci-logs/`, проверяет SHA и
timestamp запуска, описывает конкретную ошибку и не переводит исправление в
`released`.

## 6. Связь с пилотными документами

- [docs/USER_GUIDE.md](USER_GUIDE.md) описывает, какие данные участник передает
  в поддержку.
- [docs/COUNCIL_GUIDE.md](COUNCIL_GUIDE.md) определяет, когда вопрос уходит в
  Совет, HITL или go/no-go.
- [docs/FAQ.md](FAQ.md) содержит короткие ответы для участников и поддержки.
- [docs/STAGE_7_ACCEPTANCE.md](STAGE_7_ACCEPTANCE.md) фиксирует приемку
  issue #94 вместе с #91-#93 и #95.
- [docs/PILOT_RETROSPECTIVE_SCALE_PLAN.md](PILOT_RETROSPECTIVE_SCALE_PLAN.md)
  принимает P0/P1 итоги поддержки в ретроспективу и план масштабирования.

Runbook обновляется после каждого P0/P1, повторяющегося вопроса, изменения SLA,
нового legal/security gate или ретроспективы пилота.
