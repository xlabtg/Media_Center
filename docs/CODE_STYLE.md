# Гайд по стилю кода

Документ фиксирует базовые правила качества для Python-кода и инфраструктурных
изменений в НМЦ. Рабочий язык проекта — русский; код, идентификаторы, команды и
форматы внешних API остаются на английском.

## Инструменты

Baseline для новых Python-модулей:

- Python 3.13.
- `ruff` для lint, сортировки импортов и быстрой автофиксации.
- `ruff format` как основной форматтер локального CI.
- `black` как совместимый форматтер и дополнительная проверка стандарта 88
  символов.
- `mypy` в строгом режиме.
- `pytest` для unit- и контрактных тестов.
- `pre-commit` для автоматического запуска проверок до коммита.

Версии инструментов закреплены в `requirements-dev.txt`, настройки — в
`pyproject.toml`, hooks — в `.pre-commit-config.yaml`.

## Локальная установка

```bash
python -m pip install -r requirements-dev.txt
pre-commit install
```

Перед PR выполните локальный набор проверок:

```bash
ruff check .
ruff format --check .
black --check .
mypy .
pytest
```

Для ручного запуска всех pre-commit hooks:

```bash
pre-commit run --all-files
```

## Python

- Пишите код с явными типами на публичных функциях, методах и dataclass/model
  границах.
- Не используйте неявные `Any`, если тип можно выразить явно.
- Исключения должны быть доменными и пригодными для аудита; не скрывайте
  исходную причину ошибки.
- Импорты сортирует `ruff`; вручную группируйте только если это повышает
  читаемость внутри уже отсортированного блока.
- Для структурированных данных используйте Pydantic-модели, dataclass или
  типизированные словари, а не произвольные `dict[str, object]`.
- Новая бизнес-логика сопровождается минимальными unit-тестами.

## Мультитенантность и аудит

- Все сущности, события, фоновые задачи и записи аудита, относящиеся к данным
  участника или организации, содержат `tenant_id`.
- Межтенантный доступ возвращает `403` с кодом `tenant_isolation_violation` и
  фиксируется в аудите.
- В блокчейн-аудит передаются только SHA256-хэши и метаданные: без ПДн,
  денежных сумм и секретов.
- Тесты tenant-aware поведения обязательны для кода, который читает или меняет
  данные тенанта.

## Коммиты и PR

Коммиты используют Conventional Commits:

```text
<тип>(<scope>): <краткое описание>
```

Допустимые типы: `feat`, `fix`, `docs`, `test`, `refactor`, `chore`, `ci`,
`perf`. Описание пишется на русском, в повелительном наклонении и без точки в
конце.

PR заполняется по `.github/pull_request_template.md`: укажите связанный issue,
что сделано, как проверено, и отметьте безопасность/комплаенс, если изменение
затрагивает данные, платежи, аудит или внешние платформы.
