# ADR-0001: Границы микросервисов и C4 baseline

- **Статус:** Accepted
- **Дата:** 2026-06-18
- **Связанный issue:** [#5](https://github.com/xlabtg/Media_Center/issues/5)

## Контекст

НМЦ проектируется как мультитенантная платформа для медиа-кооператива с
прозрачным учётом вклада, генерацией и публикацией контента, HITL-выплатами и
неизменяемым аудитом. Система должна развиваться поэтапно: сначала
документационный и инфраструктурный фундамент, затем ключевые сервисы и
расширенные модули.

Для команды нужен общий baseline архитектуры: контекст, контейнеры,
компонентные границы и ответственность сервисов. Без этого дальнейшие задачи по
модели данных, API, тестам и инфраструктуре будут расходиться.

## Решение

Принять микросервисную архитектуру с пятью ключевыми сервисами ядра:

1. **Contribution Ledger & Weight Engine** — учёт вклада, расчёт баллов, Кв и
   долей выплат.
2. **Content Generator & Link Router (CGLR)** — генерация контента, шаблоны,
   L1/L2/L3 ссылки и логирование вклада.
3. **HITL Payout Gateway** — очередь выплат, окно вето, 2FA, коннекторы
   исполнения и audit hash.
4. **Unified Messenger Adapter** — публикация на площадках через единый
   интерфейс, трансформация контента, ретраи и реестр площадок.
5. **Private Blockchain Auditor** — запись и проверка SHA256-хэшей операций в
   приватном audit-контуре.

API Gateway становится единой точкой входа для клиентов и tenant-aware
маршрутизации. Вспомогательные модули (Policy Manager, Notification Gateway,
Analytics Engine, Activity Command Center, Voice-to-Chain, Wallet и др.)
подключаются через те же принципы контрактов и tenant-изоляции.

C4 baseline фиксируется в [ARCHITECTURE.md](../ARCHITECTURE.md): Level 1
Context, Level 2 Container и Level 3 Component для ключевых сервисов.

## Последствия

- Каждая будущая задача должна явно указывать сервис-владельца данных и
  контракт взаимодействия.
- Общие concerns (`tenant_id`, аудит, RBAC, ошибки, наблюдаемость) выносятся в
  `shared` и инфраструктурные слои, а не дублируются произвольно.
- Сервисные границы допускают независимую разработку и тестирование, но требуют
  строгого каталога REST/gRPC и RabbitMQ-контрактов.
- Монолитная реализация допустима только как временный dev scaffold, если она
  сохраняет те же границы модулей и контрактов.
