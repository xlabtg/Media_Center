# ADR-0002: Синхронный API Gateway и асинхронный RabbitMQ

- **Статус:** Accepted
- **Дата:** 2026-06-18
- **Связанный issue:** [#5](https://github.com/xlabtg/Media_Center/issues/5)

## Контекст

Платформе нужны два типа взаимодействий:

- команды и запросы, где клиенту нужен немедленный ответ, проверка RBAC и
  tenant-aware маршрутизация;
- доменные события и длительные операции: публикации, выплаты, уведомления,
  аудит, перерасчёт весов и ретраи.

Использование только синхронных вызовов сделает публикации, HITL и аудит
хрупкими. Использование только событий усложнит пользовательские команды,
авторизацию и обратную связь в интерфейсах.

## Решение

Принять гибридную модель:

- **API Gateway + REST** для внешних клиентских команд, чтения состояния,
  административных операций и внутреннего синхронного API между доверенными
  сервисами.
- **RabbitMQ** для доменных событий, фоновых задач, ретраев, уведомлений,
  статусов публикаций, HITL-переходов и audit requests.
- **gRPC** для низкоуровневого коннектора Private Blockchain Auditor к
  приватной блокчейн-сети.

Подробные правила зафиксированы в [contracts/sync-api.md](../contracts/sync-api.md)
и [contracts/events.md](../contracts/events.md).

## Последствия

- Команды должны быть идемпотентными через `Idempotency-Key`, если операция
  создаёт ресурс или запускает side effect.
- События должны содержать `event_id`, `tenant_id`, `correlation_id`, `source`,
  `type`, `occurred_at`, `schema_version` и `payload`.
- Сервисы не должны ждать RabbitMQ-события внутри HTTP request path, если это
  может привести к долгой блокировке клиента.
- Для критичных workflow нужен outbox/inbox pattern или эквивалентная
  гарантия, чтобы не терять события при сбоях между БД и RabbitMQ.
