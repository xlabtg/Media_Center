# ADR-0008: Оптимизация размера сервисных образов

- **Статус:** Accepted
- **Дата:** 2026-06-22
- **Связанный issue:** [#232](https://github.com/xlabtg/Media_Center/issues/232)

## Контекст

Эпик B Этапа 9 задает эталонный Docker-образ для сервисов НМЦ. Для B7 нужно
принять решение по базовой стратегии размера образов, зафиксировать числовой
бюджет REQ-1/REQ-N1, настроить `.dockerignore` и дать исходный замер для
будущего F2-гейта.

Текущий Dockerfile использует baseline из ADR-0006:
`python:3.13.14-slim`. Исследование Этапа 9 показывает, что `slim` дает
практичный баланс совместимости и размера, а `distroless` уменьшает поверхность
атаки, но усложняет отладку: в таком образе нет shell и привычных runtime
утилит. Для сервисов НМЦ важны воспроизводимый CI, быстрый локальный smoke-run,
healthcheck на `/health` и понятная диагностика при пилоте.

## Решение

Принять `python:3.13.14-slim` как стандартный runtime base для сервисных
образов. `distroless` допускается как опциональный профиль для чувствительных
сервисов после отдельного PR/ADR, если сервис:

- не требует shell/runtime-утилит для эксплуатации;
- имеет полноценные `/health`, `/ready`, `/info`, `/metrics` и JSON-логи;
- проходит smoke-run без интерактивной диагностики внутри контейнера;
- имеет документированный rollback на `slim`.

Для REQ-N1 принять числовые бюджеты на один сервисный runtime-образ:

| Уровень | Бюджет | Назначение |
|---------|--------|------------|
| Базовый бюджет | **< 250 МБ** | Жесткий целевой порог для F2 CI-гейта. |
| Stretch-бюджет | **stretch < 200 МБ** | Цель оптимизации после разделения сервисных зависимостей и multi-arch публикации. |

Основной показатель бюджета — локальный размер runtime-образа из `docker image
ls`/`docker system df -v`, потому что он отражает footprint после загрузки на
узел. Сжатый registry-transfer размер можно фиксировать как дополнительную
метрику, но он не заменяет локальный бюджет.

Для минимизации build context принять `.dockerignore`, исключающий VCS, CI,
документацию, тесты, эксперименты, кеши, логи, локальные env-файлы и временные
артефакты. В контексте сборки должны оставаться только файлы, которые реально
использует `infra/docker/service.Dockerfile`: `pyproject.toml`, `services/`,
`libs/`, `docker/entrypoint.sh` и сам Dockerfile.

Базовый замер и команды воспроизведения ведутся в
`docs/operations/image-size-budget.md`.

## Последствия

- F2-гейт должен сравнивать каждый сервисный образ с базовым бюджетом `< 250 МБ`
  и отдельно показывать прогресс к stretch-цели `< 200 МБ`.
- Исключения из бюджета допустимы только через отдельный ADR или risk record с
  сервисом, причиной, текущим размером и сроком пересмотра.
- Текущий замер `media-center-contribution-ledger:issue232-baseline` показывает
  превышение локального бюджета из-за установки общего набора зависимостей из
  `[project].dependencies` во все сервисные образы. Это не меняет принятый
  бюджет, но фиксирует технический долг для следующих задач B/F: разделить
  service-specific runtime dependencies или ввести иной механизм slim-сборки.
- `distroless` не становится дефолтом, чтобы не ухудшить диагностику пилотных
  сервисов раньше готовности observability и smoke-процедур.
