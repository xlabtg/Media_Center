# UX-исследование и прототипы НМЦ

Статус: baseline для issue #13. Документ готовит согласование
пользовательских сценариев, низкодетальных прототипов и дизайн-системы v0 для
этапа 4.

## 1. Область и вводные

Документ описывает первые интерфейсные решения для пилотного контура НМЦ:

- веб-кабинет пайщика;
- панель Совета для Human-in-the-Loop операций;
- входной путь аудитории в онбординг;
- основу дизайн-системы для будущего UI-кита.

Входные источники: [VISION.md](VISION.md), [REQUIREMENTS.md](REQUIREMENTS.md),
[GOVERNANCE.md](GOVERNANCE.md), [ECONOMICS.md](ECONOMICS.md),
[SECURITY.md](SECURITY.md), [COMPLIANCE.md](COMPLIANCE.md).

UX-решения подчиняются ограничениям MVP:

| Ограничение | Следствие для интерфейса |
|-------------|--------------------------|
| Human-in-the-Loop | Чувствительные действия показывают причину, окно вето, 2FA и аудит. |
| Мультитенантность | Переключение tenant-контекста не смешивает данные; каждый экран показывает текущий tenant. |
| Приватность по дизайну | ПДн раскрываются только по роли; публичные и audit-экраны используют минимум данных. |
| Кооперативная экономика | Баллы, Кв, доли и правила расчёта объясняются рядом с показателями. |
| Пилот 15-25 участников | Интерфейс оптимизирован под ручной контроль, таблицы и понятные очереди, а не под массовый enterprise-режим. |

## 2. Карта пользовательских сценариев

### 2.1. Роли и ожидания

| Роль | Основная цель | Критичный страх | UX-акцент |
|------|---------------|-----------------|-----------|
| Аудитория | Читать достоверный контент и понять, как присоединиться. | Непонятно, кто отвечает за материалы и данные. | Публичный контекст, доверие, прозрачный вход в онбординг. |
| Ассоциативный пайщик | Видеть задачи, вклад, статус и путь к действительному пайщику. | Вклад не учтён или правила непрозрачны. | Личный прогресс, история начислений, объяснение баллов. |
| Действительный пайщик | Участвовать в решениях и контроле чувствительных операций. | Выплаты или политики проходят без контроля. | Очередь вето, уведомления, ясные основания решений. |
| Совет | Управлять порогами, HITL-операциями, аудитом и спорными случаями. | Ошибка ИИ, юридический риск, межтенантная утечка. | Dense dashboard, приоритеты риска, быстрые действия с 2FA. |
| Правление | Назначать задачи, кампании, публикации и операционные настройки. | Операционная картина распадается между сервисами. | Командный центр активности, статусы, фильтры, журнал AI-действий. |
| Президиум Совета | Проверять аудит и разрешать споры. | Невозможно восстановить цепочку решений. | Timeline, audit hash, неизменяемые события, экспорт. |

### 2.3. Основные пользовательские пути

**Аудитория -> ассоциативный пайщик**

1. Пользователь читает публичный материал.
2. Открывает описание НМЦ и видит понятный вход: "присоединиться к Активу".
3. Проходит регистрацию, каналы связи и согласия.
4. Получает ассоциативный статус после подтверждения Правлением или Советом.
5. Видит первую задачу и правила начисления вклада.

**Пайщик -> учтённый вклад -> прозрачная доля**

1. Пайщик выполняет задачу или создаёт материал.
2. Событие появляется в истории вклада со статусом обработки.
3. После расчёта видны `final_points`, Кв, ограничение 0.10 и audit hash.
4. Пайщик может открыть формулу и источник события.
5. При споре создаётся обращение без раскрытия лишних ПДн другим ролям.

**Совет -> HITL-выплата -> аудит**

1. HITL Payout Gateway создаёт очередь выплат.
2. Совет видит дедлайн окна вето, риск-флаги, источник расчёта и policy version.
3. Участник Совета открывает детали, проверяет суммы в разрешённом контуре.
4. Действие подтверждается через 2FA или отклоняется с обязательной причиной.
5. Итог фиксируется в audit timeline и отправляет hash в audit-chain.

## 3. Информационная архитектура

### 3.1. Общая оболочка

| Зона | Назначение |
|------|------------|
| Верхняя панель | Текущий tenant, глобальный поиск, уведомления, профиль, индикатор роли. |
| Левая навигация | Разделы, доступные роли пользователя в текущем tenant. |
| Рабочая область | Таблицы, очереди, карточки операций, формы и audit timeline. |
| Контекстная панель | Детали выбранного объекта, риски, комментарии, связанные события. |
| Системные состояния | Loading, empty, error, forbidden, stale data, offline/retry. |

### 3.2. Навигация по ролям

| Раздел | Аудитория | Пайщик | Совет | Правление | Президиум |
|--------|-----------|--------|-------|-----------|-----------|
| Публичные материалы | Да | Да | Да | Да | Да |
| Онбординг | Да | Да | Нет | Да | Нет |
| Мой вклад | Нет | Да | Да, read-only по правам | Да | Да |
| Задачи | Нет | Да | Да | Да | Да |
| Выплаты | Нет | Свои статусы | HITL и вето | Очереди | Аудит |
| Политики | Нет | Read-only | Да | Draft | Audit |
| Аудит | Нет | Только свои события | Да | Да | Да |
| Аналитика KPI | Нет | Ограниченно | Да | Да | Да |

### 3.3. Приоритет экранов MVP

1. Кабинет пайщика: вклад, баланс, задачи, история.
2. Панель Совета: очередь HITL, вето, подтверждения, пороги.
3. Activity Command Center: участники, недельная активность, кампании.
4. Audit timeline: события, hash, policy version, комментарии.
5. Онбординг: регистрация, согласия, статус, первая задача.

## 4. Wireframes v0

Ниже показаны low-fidelity wireframes. Они фиксируют состав экранов и
приоритет информации, но не задают финальную визуальную стилистику.

### 4.1. Веб-кабинет пайщика

```text
+----------------------------------------------------------------------------+
| НМЦ / tenant: Пилотный кооператив      Поиск      Уведомления  Профиль    |
+----------------------+-----------------------------------------------------+
| Мой обзор            | Вклад за неделю        Кв             Статус        |
| Задачи               | 14.5 балла             0.073          ассоц. пайщик |
| Вклад и баланс       +-----------------------------------------------------+
| История              | Следующая лучшая задача                              |
| Голосования          | [Кампания: Медиацентр] Написать тезисы к материалу   |
| Настройки согласий   | Дедлайн: завтра 18:00       [Взять задачу]           |
|                      +-----------------------------------------------------+
|                      | Динамика вклада                                     |
|                      | [график баллов]      Формула: base * platform * ... |
|                      +-----------------------------------------------------+
|                      | История событий                                     |
|                      | Дата     Тип       Баллы   Статус       Audit hash  |
|                      | 18.06    publish   5.0     учтено       sha256:...  |
|                      | 17.06    idea      0.5     проверка     sha256:...  |
+----------------------+-----------------------------------------------------+
```

Ключевые решения:

- первый экран отвечает на вопрос "что мне делать дальше";
- баллы, Кв и статус показываются рядом, чтобы пайщик видел связь вклада и
  прав;
- история событий содержит статус обработки и audit hash;
- спор по событию открывается из строки истории, без отдельного поиска;
- на мобильном экране левая навигация сворачивается в нижние вкладки:
  "Обзор", "Задачи", "Вклад", "Ещё".

Состояния:

| Состояние | Поведение |
|-----------|-----------|
| Нет событий | Показать первую задачу и краткое правило начисления. |
| Расчёт задерживается | Показать статус "в обработке" и ожидаемое время обновления. |
| Нет прав | Не скрывать раздел полностью; показать, какая роль нужна и где запросить доступ. |
| Спор по начислению | Открыть форму с выбранным событием, причиной и вложениями без ПДн в audit-chain. |

### 4.2. Панель Совета

```text
+----------------------------------------------------------------------------+
| НМЦ / tenant: Пилотный кооператив       Роль: Совет      2FA active        |
+----------------------+---------------------------+-------------------------+
| Очередь HITL         | Очередь выплат            | Детали операции         |
| Вето                 | [HIGH] Выплата #P-104     | ID: P-104               |
| Политики             | Окно вето: 05:12:44       | Policy: payout-v3       |
| Аудит                | Участников: 12            | Риск: новый получатель  |
| KPI                  | [Открыть]                 |                         |
|                      |---------------------------| Расчёт                  |
|                      | [MED] Изменение порога    | Пул: доступен по роли   |
|                      | Ожидает 2 голоса          | Кв: capped at 0.10      |
|                      | [Открыть]                 |                         |
|                      |---------------------------| Audit timeline          |
|                      | [LOW] Публикация кампании | queued -> reviewed      |
|                      | Требует просмотра         | sha256:...              |
|                      | [Открыть]                 |                         |
|                      |                           | [Вето] [Подтвердить]    |
+----------------------+---------------------------+-------------------------+
```

Ключевые решения:

- очередь сортируется по риску и дедлайну окна вето;
- карточка операции показывает policy version, источник расчёта и audit trail;
- действия "Вето" и "Подтвердить" всегда требуют явного подтверждения, а
  финансовые операции требуют 2FA;
- причина вето обязательна и попадает в журнал;
- детали сумм и ПДн видны только ролям с правом доступа, audit-chain хранит
  только hash и метаданные.

Состояния:

| Состояние | Поведение |
|-----------|-----------|
| Окно вето истекает | Закрепить операцию наверху и усилить уведомление Совета. |
| Нет 2FA | Заблокировать подтверждение, оставить просмотр и комментарии. |
| Конфликт голосов | Показать текущий кворум, историю решений и следующий допустимый шаг. |
| Ошибка внешнего шлюза | Отложить исполнение, сохранить решение Совета и показать retry status. |

### 4.3. Онбординг аудитории

```text
+-------------------------------------------------------------+
| Добро пожаловать в НМЦ                                      |
| 1. Контакты  2. Согласия  3. Профиль  4. Первая задача      |
+-------------------------------------------------------------+
| Каналы связи                                                |
| Телефон        [________________]                           |
| Telegram       [________________]                           |
| Email          [________________]                           |
|                                                             |
| Согласия                                                    |
| [ ] Обработка данных для участия в tenant                   |
| [ ] Уведомления о задачах и HITL                            |
| [ ] Аналитика вклада без публичного раскрытия ПДн           |
|                                                             |
|                              [Назад] [Продолжить]           |
```

Онбординг должен быть коротким: аудитория не обязана понимать всю экономику до
регистрации. Полные правила раскрываются в кабинете после получения статуса.

## 5. Дизайн-система v0

### 5.1. Принципы

| Принцип | Практическое правило |
|---------|----------------------|
| Рабочая плотность | Интерфейсы похожи на операционный инструмент: таблицы, очереди, фильтры, контекстные панели. |
| Проверяемость | У важных действий есть источник, policy version, actor, timestamp и audit hash. |
| Человеческий контроль | AI-рекомендации отделены от финальных решений человека. |
| Минимизация ПДн | По умолчанию показываются статусы и агрегаты, детали раскрываются по роли. |
| Стабильные размеры | Таблицы, кнопки и карточки не меняют размер от статусов, таймеров и длинных hash. |

### 5.2. Цветовые токены

| Токен | Значение | Назначение |
|-------|----------|------------|
| `color.bg.canvas` | `#F6F7F9` | Фон приложения. |
| `color.bg.surface` | `#FFFFFF` | Рабочие области, таблицы, формы. |
| `color.text.primary` | `#111827` | Основной текст. |
| `color.text.secondary` | `#4B5563` | Подписи, метаданные, вторичный текст. |
| `color.border.default` | `#D8DEE6` | Разделители и границы. |
| `color.brand.primary` | `#1F6F8B` | Основные действия и активная навигация. |
| `color.brand.accent` | `#C45A2A` | Акцент для важных, но не опасных действий. |
| `color.state.success` | `#2F855A` | Успешная обработка, подтверждение. |
| `color.state.warning` | `#B7791F` | Истекающее окно, ручная проверка. |
| `color.state.danger` | `#C53030` | Вето, отказ, критичный риск. |
| `color.state.info` | `#2B6CB0` | Нейтральная информация и ссылки. |
| `color.focus` | `#6B46C1` | Видимый focus-ring для клавиатурной навигации. |

Цвета статусов не должны быть единственным способом передачи смысла: рядом
используются текстовый статус, иконка или короткий код риска.

### 5.3. Типографика

| Токен | Размер / line-height | Назначение |
|-------|----------------------|------------|
| `font.family.base` | system UI stack | Основной интерфейс. |
| `font.family.mono` | monospace | Hash, ID, технические значения. |
| `font.size.xs` | 12 / 16 | Метаданные, подписи таблиц. |
| `font.size.sm` | 14 / 20 | Основной текст плотных интерфейсов. |
| `font.size.md` | 16 / 24 | Формы, описания, модальные окна. |

Базовый интерфейс не использует hero-типографику. В операционных экранах важнее
сканируемость таблиц, чем крупные декоративные заголовки.

### 5.5. Компоненты v0

| Компонент | Назначение | Обязательные состояния |
|-----------|------------|------------------------|
| `AppShell` | Верхняя панель, роль, tenant, навигация. | active, collapsed, forbidden. |
| `MetricTile` | Баллы, Кв, KPI, таймеры. | normal, warning, stale, loading. |
| `DataTable` | История вклада, участники, выплаты, аудит. | sort, filter, empty, error. |
| `StatusBadge` | Статус задачи, операции или риска. | success, warning, danger, info. |
| `TaskCard` | Следующее действие пайщика. | available, assigned, overdue, done. |
| `HITLQueueItem` | Элемент очереди Совета. | low, medium, high, expired. |
| `VetoActionBar` | Вето/подтверждение чувствительной операции. | disabled, needs-2fa, confirm, veto. |
| `TwoFactorDialog` | Подтверждение выплат и критичных политик. | input, invalid, locked, success. |
| `AuditHash` | Отображение и копирование hash. | copied, truncated, verified, mismatch. |
| `Timeline` | Цепочка решений и событий. | queued, reviewed, approved, rejected. |
| `ConsentControl` | Согласия онбординга. | required, optional, revoked. |
| `EmptyState` | Пустые разделы без декоративной перегрузки. | first-task, no-results, no-access. |
| `InlineAlert` | Ошибка, предупреждение, юридический риск. | info, warning, danger, success. |

### 5.6. Доступность и UX-качество

- Все действия доступны с клавиатуры, focus state видим.
- Минимальный контраст текста к фону: 4.5:1 для обычного текста.
- Таймер окна вето сопровождается абсолютным временем дедлайна.
- Длинные ID и hash обрезаются визуально, но копируются полностью.
- Таблицы имеют явные пустые, загрузочные и ошибочные состояния.
- Опасные действия требуют подтверждения с кратким объяснением последствия.

1. Нужно ли в пилотном веб-кабинете показывать прогноз выплаты или только долю
   и статус очереди?
2. Какие операции, кроме выплат и изменения политик, должны сразу попадать в
   высокий риск в панели Совета?
3. Должен ли ассоциативный пайщик видеть голоса Совета по своему спору или
   только итоговое решение и обоснование?
4. Требуется ли отдельный публичный профиль автора на этапе MVP или достаточно
   карточки материала и онбординга?

## 8. Критерии готовности для этапа 4

- Сценарии пайщика, Совета и аудитории согласованы в PR.
- Wireframes веб-кабинета и панели Совета покрывают первый экран, основное
  действие, ошибку доступа и пустое состояние.
- Дизайн-система v0 содержит токены цвета, типографики, отступов и перечень
  компонентов.
- Все чувствительные действия имеют явный Human-in-the-Loop паттерн.
- UX-решения не требуют раскрытия ПДн сверх прав роли и tenant-контекста.
